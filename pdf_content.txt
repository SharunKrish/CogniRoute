A practical 48-hour assessment to separate real builders from resume polishers. Candidates must
build a focused backend-heavy product that receives customer requests, routes them through an AI
workflow, assigns priority, tracks status, and exposes realtime updates to an admin dashboard.

This is not a UI-only assignment. We are evaluating backend architecture,
database design, async processing, AI workflow thinking, reliability, security

Main Skill Signal

Can the candidate design and ship a real workflow system under constraints?

Candidate Brief
Build a mini AI-powered customer request routing system. The app should let a business receive
customer requests, classify them using AI or a mock AI workflow, assign priority, track status, and
show realtime updates in a simple admin dashboard.
The system should feel like a small internal operations tool that could actually be used by a support,
sales, or automation team.

Scenario
Cognifyr receives customer messages from multiple channels such as website forms, WhatsApp, and
email. Each request must be stored, classified, prioritized, routed into a workflow, and tracked by
admins until it is resolved.

1 A customer message enters the system through an API or webhook simulation.
2 The request is saved immediately and marked as queued/new.
3 A background worker classifies the request using AI or a mock AI service.
4 The system stores the AI classification separately from the original request.
5 The admin dashboard receives a live update and shows the latest status.
6 An admin or agent can update status and add internal notes.

Required Build Scope
1

Backend API
User/admin auth, create request, list/filter requests, view details, update status,
add internal notes. REST API is required.

2

Database Design
Design tables/collections for users, customer_requests, request_events,
ai_classifications, and internal_notes. Explain schema choices.

3

AI Classification Workflow
When a request is created, classify category, assign priority, generate a short
summary, and store AI output separately. Mock AI is acceptable.

4

Async / Background Processing
AI classification must not block the request creation API. Use BullMQ + Redis, a
worker, Celery, or a clearly explained queue simulation.

5

Realtime Updates
Admin dashboard should receive live updates using WebSocket, Socket.io,
Server-Sent Events, Supabase realtime, Firebase, or equivalent.

6

Security
JWT/session auth, password hashing, validation, rate limiting or abuse protection,
environment variables, and no secrets in the frontend or repository.

7

Frontend Dashboard
Simple but usable admin dashboard with login, request list, request detail, filters,
notes, status update, and realtime status/classification changes.

8

Deployment
Deploy on Vercel, Render, Railway, Fly.io, Cloudflare, Supabase, Firebase, VPS, or
provide a clear demo video with local setup.

MVP Principle
A clean, working MVP is better than an unfinished enterprise system. The project should prove that
the candidate can design workflows, write maintainable backend code, and explain technical
tradeoffs clearly.

AI Workflow and Skills Layer
The AI layer should be designed like a replaceable skill/module. Candidates can use a real provider if
they have access, but the assessment must be possible with a clean mock provider.

Classification
Skill

Classifies the request into sales, support, urgent, spam, or
other. It should return structured output, not random text.

Priority Skill

Assigns low, medium, or high priority based on message
urgency, intent, customer language, and business impact.

Summary Skill

Generates a short internal summary that helps an admin
understand the request quickly.

Routing Skill

Decides which queue or workflow should handle the request.
This can be simple, but it should be clearly explained.

Expected AI Output Shape

Important: Real API access is not required. A well-designed mock AI layer is better

Security, Reliability, and Delivery
Rules
Required Security Basics
●

JWT or session authentication for protected admin APIs.

●

Password hashing using bcrypt, argon2, or platform-auth equivalent.

●

Input validation for request creation, login, filters, status updates, and notes.

●

Rate limiting or basic abuse protection on public endpoints.

●

Environment variables for secrets and provider API keys.

●

No API keys, tokens, passwords, or private data committed to GitHub.

●

Basic explanation of webhook/API signature validation for inbound channels.

Reliability Expectations
●

Failed AI classification should not break request creation.

●

The system should track failed jobs and expose status clearly.

●

Retry logic is a bonus but failure states should still be visible.

●

Events should be recorded so reviewers can understand what happened and when.

Submission Deliverables
1 GitHub repository link with clean commit history.
2 Live demo link or 3-5 minute demo video.
3 README with setup instructions, environment variables, architecture explanation, schema
explanation, AI workflow explanation, and known limitations.

4 API documentation.
5 Architecture diagram.
6 Short tradeoff note: "What I would improve with two more weeks."

Important Rules
●

Do not copy-paste a complete project from the internet.

●

Using AI tools is allowed, but the candidate must understand and explain the architecture.

●

Real AI API usage is optional. Mock AI is acceptable if the architecture is clean.

●

Keep the project focused. A clean working system is better than an unfinished overbuilt system.
